## Как переименовать ветку?

Убедимся что находимся в нужной ветке:
git branch --all

Меняем имя ветки:
git branch -m старое_имя_ветки новое_имя ветки

Сопоставляем новое имя локальной ветки с удаленной:
git push -u origin новое_имя_ветки

Удаляем старую ветку:
git push origin --delete старое_имя_ветки


### Как удалить файл/папку из репозитория?

git rm /путь_к_файлу
git rm -r /путь_к_каталогу

если дрогнула рука и удалили не тот файл\каталог
> git restore имя_файла
если рука дрогнула слишком сильно, и закомитила удаление файла, можно восстановить его из последнего коммита
> git restore --source HEAD -- имя_файла
или из определенного комита с указанием хэша
> git restore --sourse хэш_коммита -- имя файла

### Как объединить несколько коммитов в один?

> git rebase -i HEAD~n
 (где n - количество коммитов, идущих друг за другом, которые выхотите объединить в один)
далее, в открывшемся редакторе будет отображен список из n коммитов в порядке от более раннего, к более позднему
нужно изменить начало строки всех коммитов, кроме самого раннего с pick на s, сохранить и закрыть редактор
после чего откроется следующий редактор, в котором следует указать описание получвшегося коммита

### Для чего нужна команда git bisect?

Позволяет производить поиск коммита, в котором было что либо изменено, путем бинарного поиска.
На примере ошибки: указываем номер "плохого" и "хорошего" коммита. Git переключает нас на коммит, находящийся в середине списка между "плохим" и "хорошим" коммитом, для проверки воспроизводимости ошибки. Таким образом, за каждую итерацию можно отсекать половину списка. Поиск может производится не только по ошибкам, но и по любым изменениям в коде.

### Как настроить проверку кода перед коммитом?

Первый способ:
помещаем интересующие нас хуки в /каталог_проекта/.git/hooks/pre-commit
убеждаемся что права на выполнение присутствуют

Второй способ:
создаем каталог в котором будут хранится наши хуки и добавляем в конфиг git:
git config --global core.hooksPath /каталог_с_хуками

### Что такое тэги в репозитории?

Особый указатель на коммит, использующийся в основном для обозначения релиза версии проекта. Не заменяет хэш коммита, не является веткой. Позволяет легче ориентироваться среди версий проекта, легче откатываться к предыдущему релизу, если после выкатки обновления обнаружились баги.
Существует два вида тегов:
- аннотированные  - несут в себе дополнительную информацию об авторе тега и дате его создания, а так же аннотацию к нему.
- легковесные - не содержат в себе дополнительной информации.

### Как поменять ссылку на удаленный репозиторий?

> git remote set-url origin новая_ссылка

### Как можно отменить неопубликованный коммит?

В зависимости от того, какой результат хотим получить:
- git reset HEAD^ - откатит к предыдущему коммиту, уберет изменения из стейджинга, но не изменит файлы в каталоге.
- git reset --soft HEAD^ - откатит к предыдущему коммиту, но сохранит изменения в стейджинге.
- git reset --hard HEAD^ - откатит к предыдущему коммиту, все внесенные изменения будут утеряны.

### Чем отличается git fetch от git pull?

git fetch загружает в локальный репозиторий(.git/refs/remotes/) изменения из удаленного, не производит слияние веток, что позволяет нам:
-сравнить локальную и удаленную ветки
-посмотреть историю коммитов
-переключиться на обновленную удаленную ветку
После чего на основании этой информации мы решим будем ли производить слияние или нет

git pull по сути выполняет последовательно git fetch и затем git merge, сразу применяя все изменения в локальной ветке.
